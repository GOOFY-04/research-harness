● 现在我对你的背景有了清晰的认识。agent4d 是一个针对 3DGS 的单脚本式研究
  agent，而你现在想做的是一个通用的、面向长流程科研的 harness 框架。

  这两者的核心差距在于：

  agent4d 的问题
  - 单脚本线性执行，中断就得重来
  - 没有状态机，无法从断点恢复
  - agent 逻辑和业务逻辑耦合在一起
  - 无法协调多个专职 sub-agent

  一个真正的科研 harness 需要什么

  长流程科研 = 多阶段 × 可中断 × 可恢复 × 多 agent 协作

  阶段示例：
    选题 → 文献调研 → 方法设计 → 实验规划 → 代码实现
         → 实验执行 → 结果分析 → 论文撰写 → 审稿迭代

  核心设计要解决三个问题：

  1. 状态持久化 — 每个阶段完成后 checkpoint，进程崩溃可从上次恢复
  2. 工作流引擎 — 用声明式配置定义阶段依赖、条件分支、重试策略
  3. Agent 解耦 — 文献 agent、代码 agent、写作 agent 各司其职，通过统一接口通信

  ---
  建议的工程结构：

  research-harness/
  ├── harness/
  │   ├── core/
  │   │   ├── workflow.py      # 工作流引擎 + 状态机
  │   │   ├── checkpoint.py    # 断点续跑
  │   │   ├── agent.py         # BaseAgent 抽象
  │   │   └── memory.py        # 跨 session 记忆
  │   ├── agents/
  │   │   ├── planner.py       # 选题 & 规划
  │   │   ├── literature.py    # 文献调研
  │   │   ├── coder.py         # 代码实现
  │   │   ├── reviewer.py      # 自我审稿
  │   │   └── writer.py        # 论文撰写
  │   └── tools/               # 工具层（arxiv、代码执行、通知等）
  ├── workflows/
  │   └── research.yaml        # 声明式流程定义
  ├── sessions/                # 每次运行的状态存储
  ├── configs/
  │   └── default.yaml
  └── main.py
